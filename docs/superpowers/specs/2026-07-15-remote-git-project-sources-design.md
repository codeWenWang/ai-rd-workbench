# GitHub 与 Gitee 项目接入设计

## 目标

在现有本地项目分析能力上增加公开 GitHub 和公开 Gitee 仓库接入。用户输入仓库地址后，系统将源码浅克隆到本机受控缓存目录，再复用现有的扫描、项目问答、架构图、流程图、时序图和接口文档流程。

首版优先保证流程完整、可实际运行，不加入账号登录、Token、私有仓库、Webhook 或自动定时同步。

## 支持范围

- 支持 `https://github.com/<owner>/<repo>` 和带 `.git` 后缀的地址。
- 支持 `https://gitee.com/<owner>/<repo>` 和带 `.git` 后缀的地址。
- 只使用仓库默认分支。
- 只读取公开仓库，不保存 GitHub 或 Gitee 凭据。
- 保持现有本地目录接入方式不变。
- 不执行仓库代码、安装依赖、运行构建脚本或向远程仓库写入内容。

## 方案选择

采用本机 Git CLI 进行浅克隆和更新：

- 首次连接执行单分支、无标签、深度为 1 的浅克隆。
- 重新扫描前执行快进更新。
- GitHub 与 Gitee 使用同一套 Git 能力，仅在 URL 校验和来源展示上区分供应商。

不采用 ZIP 下载，因为它缺少可靠的增量更新和提交信息；不采用平台 Contents API，因为大量文件请求容易受限流和网络波动影响。

## 架构

新增 `RemoteGitRepositoryManager`，职责仅包括：

1. 校验并规范化远程仓库 URL。
2. 识别 `github` 或 `gitee` 来源。
3. 计算稳定的本地缓存路径。
4. 首次浅克隆仓库。
5. 后续尝试快进更新。
6. 在删除远程项目时安全清理受控缓存。

项目应用层继续负责项目记录，项目分析层在扫描远程项目之前调用仓库管理器准备本地源码。准备完成后仍使用现有 `LocalProjectScanner`，避免复制解析、索引和制品生成逻辑。

```text
添加项目
  -> 本地目录：验证目录 -> 保存项目
  -> GitHub/Gitee：验证 URL -> 浅克隆到缓存 -> 保存项目

扫描项目
  -> 本地目录：直接扫描
  -> GitHub/Gitee：尝试更新缓存 -> 扫描缓存
      -> 更新失败且缓存完整：使用缓存并返回警告
      -> 缓存缺失或损坏：停止并返回错误
```

## 数据模型

`projects` 增加可空的 `source_uri` 字段：

- 本地项目：`source_type=local`，`root_path` 为用户目录，`source_uri` 为空。
- GitHub 项目：`source_type=github`，`root_path` 为本地缓存目录，`source_uri` 为规范化仓库 URL。
- Gitee 项目：`source_type=gitee`，其余规则与 GitHub 相同。

数据库迁移只新增可空字段，不修改现有项目、对话和分析数据。

缓存目录默认使用 `backend/data/git-projects`。单个仓库目录名由供应商、仓库名称和规范化 URL 哈希组成，避免路径冲突。缓存目录继续由 `.gitignore` 排除。

## URL 与命令安全

- 只允许 HTTPS。
- 只允许 `github.com` 和 `gitee.com`。
- 路径必须恰好包含所有者和仓库名。
- 拒绝用户名密码、端口、查询参数、片段、SSH、`file://` 和本地路径。
- Git 命令使用参数数组调用，不启用 shell 字符串拼接。
- 错误信息不得回显环境变量或本机敏感路径之外的凭据；首版没有远程凭据。
- 删除缓存前必须确认目标路径位于受控缓存根目录内，绝不删除用户的本地项目目录。

## 网络与离线行为

首次添加远程项目必须成功克隆，否则不创建项目记录。

重新扫描时：

- 更新成功：分析最新默认分支内容。
- 更新失败但本地缓存包含有效 Git 仓库：继续扫描缓存，返回 `remote_update_unavailable` 警告。
- 缓存不存在或不是有效仓库：返回明确错误，提示重新连接项目。

Git 命令继承用户当前 Git 网络和代理配置，因此已配置的 `http.proxy` 同时适用于 GitHub 和 Gitee。克隆和更新分别设置超时，防止请求无限挂起。

## API

现有 `POST /api/projects` 保持兼容，并扩展请求字段：

```json
{
  "name": "可选显示名称",
  "source_type": "local | github | gitee",
  "root_path": "本地项目绝对路径，仅 local 使用",
  "repository_url": "公开仓库 HTTPS 地址，仅 github/gitee 使用"
}
```

响应中的项目增加 `source_uri`。扫描响应继续返回文件和接口数量，并可增加远程更新警告。

## 前端

“连接项目”对话框增加来源选择：

- 本地目录
- GitHub
- Gitee

选择本地目录时显示绝对路径输入框；选择 GitHub 或 Gitee 时显示仓库地址输入框。提交成功后自动切换到新项目的项目总览，用户点击“扫描项目”即可进入现有分析流程。

项目总览副标题对本地项目显示本地路径，对远程项目显示仓库 URL，并标明 GitHub 或 Gitee 来源。其余导航和分析页面不增加新流程。

## 错误处理

对用户使用中文区分以下错误：

- 仓库地址格式不正确。
- 仓库来源与选择的平台不匹配。
- Git 未安装或无法执行。
- 仓库不存在或不是公开仓库。
- 网络连接或代理失败。
- 克隆超时或更新超时。
- 本地缓存损坏或丢失。
- 缓存目录没有写入权限。

克隆失败时清理未完成的临时目录。更新失败但使用旧缓存时不把项目标记为失败。

## 测试

后端测试覆盖：

- GitHub 和 Gitee URL 规范化与来源识别。
- 非 HTTPS、错误域名、额外路径、查询参数和凭据地址被拒绝。
- 首次克隆生成正确的安全 Git 命令和缓存目录。
- 更新成功、网络失败回退缓存、缓存损坏失败。
- 删除远程项目只删除受控缓存，本地项目目录不受影响。
- 项目 API 同时兼容旧的本地请求和新的远程请求。
- 远程项目扫描继续产生文件、路由和项目检索数据。

前端测试覆盖来源选择、条件字段、API 请求结构和远程来源展示。

最终浏览器验收至少连接一个公开 GitHub 仓库和一个公开 Gitee 仓库，完成添加、扫描、项目问答和删除。若测试环境无法访问 GitHub，GitHub 自动测试仍通过本地可控测试，实际联网验收明确记录网络限制。

## 验收标准

1. 用户可以通过公开 GitHub URL 添加项目并扫描。
2. 用户可以通过公开 Gitee URL 添加项目并扫描。
3. 两类远程项目可以使用现有问答与四类项目分析功能。
4. 网络更新失败时，已经克隆的项目仍可分析旧缓存并收到警告。
5. 删除远程项目会清理缓存，不会影响远程仓库或本地项目目录。
6. 现有本地项目接入和全部自动化测试保持正常。

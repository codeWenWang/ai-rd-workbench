# 本地多项目研发智能体设计

## 1. 目标

在现有本地单人研发知识工作台上增加项目级智能分析能力。用户可以连接多个本地项目目录，在不修改源码、不执行项目代码的前提下完成项目问答、架构图、流程图、时序图和接口文档生成，并在 DashScope 与 OpenAI 兼容模型之间切换或对比回答。

本次保持模块化单体架构：仍使用一个 FastAPI 服务、一个前端应用、SQLite 主数据源和 Pinecone 语义索引，不引入 Redis、Celery、PostgreSQL 或独立 Worker 服务。

## 2. 已确认范围

### 2.1 第一版包含

- 添加、查看、切换和删除多个本地项目。
- 通过本地绝对路径连接项目，源码目录严格只读。
- 支持 Python、FastAPI、原生 JavaScript、HTML 和 CSS 项目的基础分析。
- 手动扫描和重新扫描项目，通过文件哈希识别变化。
- 提取文件、模块、类、函数、导入、路由、调用和依赖关系。
- 生成 Mermaid 架构图、流程图和时序图。
- 生成 Markdown 项目接口文档。
- 在当前项目范围内进行 RAG 问答，并引用源码文件和行号。
- 支持 DashScope 和通用 OpenAI 兼容聊天模型。
- 支持单模型切换和最多两个模型的并行效果对比。
- 实现模型 Token 到达即发送的真实 SSE 流式输出。
- 重组侧边栏，删除当前平台 API 文档页面，将运行状态移到底部。

### 2.2 第一版不包含

- GitHub 仓库接入，但保留项目源适配接口。
- ADR 技术决策记录生成。
- 修改、格式化或提交被分析项目的代码。
- 执行被分析项目的程序、测试、构建脚本或安装脚本。
- 持续监听目录变化；第一版由用户手动触发重新扫描。
- Java、Spring Boot、Vue、React 等额外技术栈的深度解析。
- 切换 Embedding 模型；更换 Embedding 需要重建索引，后续作为高级功能处理。
- 对动态调用关系作完全准确的承诺。

## 3. 总体架构

```text
Frontend
  -> Project API / Analysis API / Model API / Chat SSE API
      -> Application Use Cases
          -> ProjectSource
              -> LocalDirectorySource
              -> GitHubRepositorySource（后续）
          -> ProjectScanner
          -> ParserRegistry
              -> PythonAstParser
              -> JavaScriptModuleParser
              -> HtmlReferenceParser
          -> ProjectAnalysisEngine
          -> ArtifactGenerator
          -> ProjectRetriever
          -> ModelGateway
      -> SQLite / Project FTS / Pinecone / LocalSecretVault
```

静态分析负责产生可核对的项目事实，大模型只基于这些事实进行摘要、组织和解释。禁止把完整项目直接交给模型后让模型凭空推断结构。

## 4. 项目源与只读边界

定义统一的 `ProjectSource` 接口，至少提供：

- 获取项目标识和显示名称。
- 枚举项目内文件。
- 读取指定相对路径的文件内容。
- 计算当前源码版本标识。

第一版实现 `LocalDirectorySource`。项目记录保存规范化后的绝对路径，但所有展示和索引只使用相对路径。

扫描器必须执行以下保护：

- 确认目录存在且可读。
- 拒绝把普通文件当作项目目录。
- 解析后的文件路径必须仍位于项目根目录内。
- 不跟随指向项目根目录外的符号链接或目录联接。
- 默认忽略 `.git`、`.venv`、`venv`、`node_modules`、`dist`、`build`、缓存和测试产物。
- 默认忽略 `.env*`、证书、私钥、凭据文件和已知敏感文件名。
- 跳过二进制文件、超大文件和不支持的扩展名。
- 不向项目目录写入标记文件、分析结果或缓存。

第一版前端使用路径输入框添加项目。浏览器目录上传会复制整个项目，不能表示持续连接关系，因此不作为主方案。

## 5. 数据模型

在现有 SQLite 中新增：

- `projects`：项目名称、源类型、根路径、源码版本、技术栈、状态和扫描时间。
- `project_files`：相对路径、语言、大小、修改时间、内容哈希和解析状态。
- `project_symbols`：模块、类、函数、方法、路由等符号及行号。
- `project_relations`：导入、依赖、调用、包含和路由到处理器等关系。
- `project_chunks`：用于项目代码全文检索和向量索引的文本分块。
- `project_analysis_jobs`：扫描、解析、索引和制品生成任务及错误信息。
- `analysis_artifacts`：制品类型、Mermaid 或 Markdown 内容、来源版本、模型和生成状态。
- `model_providers`：供应商类型、名称、Base URL、密钥引用和启用状态。
- `model_profiles`：模型名称、显示名称、参数和流式能力。

现有 `conversations` 增加可空的 `project_id` 和默认模型配置。现有通用对话可继续使用，项目对话必须绑定项目。

项目代码采用独立的 `project_chunks` 与项目 FTS 索引，避免改动现有知识库和长期记忆的所有权约束。

## 6. 扫描与静态分析

扫描流程：

```text
验证项目路径
-> 枚举和过滤文件
-> 比较文件哈希
-> 删除失效文件的分析数据
-> 解析新增或变化文件
-> 重建受影响的关系
-> 更新 FTS 和 Pinecone
-> 计算项目源码版本
```

Python 使用标准库 `ast` 提取导入、定义、调用和装饰器。FastAPI 路由通过装饰器和路由对象识别，不导入或执行目标项目。

JavaScript 第一版提取 ES Module 的 `import`、`export` 和常见函数结构。HTML 提取脚本、样式和静态资源引用。无法可靠识别的动态调用保留为未知关系，不伪造确定结果。

项目源码版本使用所有已纳入扫描文件的相对路径和内容哈希计算。后续 GitHub 项目可直接使用提交 SHA 实现同一语义。

## 7. 分析制品

### 7.1 架构图

根据目录、模块、导入和分层关系生成 Mermaid `flowchart`。默认提供项目总览，并允许按模块查看局部图。

### 7.2 流程图

以 FastAPI 路由或主要入口为起点，沿路由、应用用例、仓储和外部服务关系生成处理流程。无法确定的分支标记为“推断”。

### 7.3 时序图

按调用方向生成 Mermaid `sequenceDiagram`，展示客户端、API、应用用例、数据库、向量库和模型服务之间的交互。动态分派导致的不确定调用需要附置信提示。

### 7.4 接口文档

从静态提取的 FastAPI 路由、请求模型和响应线索生成 Markdown 文档，至少包含方法、路径、处理器、源码位置、参数和说明。若信息不足，字段显示“静态分析未确认”，不能猜测参数契约。

所有制品保存来源源码版本。项目重新扫描后，如源码版本变化，旧制品标记为 `stale`，由用户重新生成。

## 8. 项目级 RAG

每个项目使用独立 Pinecone namespace，例如 `project_<project_id>`，从存储层保证项目向量不会混用。

项目问答检索顺序：

1. 当前项目代码 FTS。
2. 当前项目 Pinecone 语义检索。
3. 与当前项目关联的知识文档。
4. 用户已确认的长期记忆。
5. 融合、重排并生成带源码路径和行号的引用。

Pinecone 不可用时，项目问答降级到项目 FTS，并向前端返回明确警告。新增或重建语义索引失败时，项目状态不能显示为完全索引成功。

## 9. 多模型网关

定义统一 `ChatModelProvider` 和 `ModelGateway`。第一版提供：

- DashScope 适配器。
- OpenAI 兼容适配器，允许配置 Base URL、API Key 和模型名称。

模型切换只影响聊天生成，不改变项目 Embedding。对比模式使用同一份系统提示、项目检索上下文和用户问题，并发调用最多两个模型，分别显示响应、延迟、引用和错误。

密钥由后端本地密钥仓库保存。SQLite 只保存密钥引用，API 不返回密钥明文，日志和异常不得包含密钥。密钥文件和加密材料放在 `backend/data`，并由 `.gitignore` 排除。该设计防止误提交和前端泄露，不用于防御已取得本机账户权限的攻击者。

## 10. 真实流式输出

当前接口先生成完整答案再切片发送，需要改为模型流直接驱动 SSE。

事件协议：

- `session`：会话与消息 ID。
- `stage`：检索、分析、生成和收尾阶段。
- `token`：模型增量文本，可携带模型 ID。
- `citations`：最终引用。
- `warning`：降级或不确定信息。
- `done`：最终状态、耗时和完整结果元数据。
- `error`：结构化错误。

用户消息和待生成助手消息在模型调用前落库。Token 在内存中聚合，成功后一次性保存完整助手内容。客户端断开或用户取消时保存已生成部分，并将消息标记为 `cancelled`，不遗留永久 `pending` 状态。

## 11. 前端信息架构

侧边栏顶部放置项目选择器和新对话按钮。主要功能为：

- 项目总览。
- 项目问答。
- 项目分析，可展开显示架构图、流程图、时序图和接口文档。
- 知识库。
- 个人记忆。
- 最近对话。

底部放置模型设置、主题切换和服务状态。运行状态不再占用主要导航项，点击底部状态后打开诊断抽屉。

删除现有平台 API 文档入口和 `/docs` 页面。保留 `/openapi.json` 用于程序、测试和开发工具。项目分析中的“接口文档”只展示当前连接项目的接口资料。

聊天顶部提供当前模型选择。模型对比采用明确开关，开启后最多选择两个模型并排展示，普通聊天保持单列界面。

## 12. 任务与错误处理

项目扫描、解析、索引和制品生成采用进程内后台任务，不阻塞 HTTP 请求。任务状态持久化到 SQLite。服务重启时，仍处于 `running` 的任务标记为 `interrupted`，允许用户重试。

主要错误必须区分：路径无效、无权限、敏感目录、解析失败、Pinecone 不可用、模型认证失败、模型不支持流式、任务被取消和源码版本已变化。

单个文件解析失败不能中断整个项目扫描；任务完成后展示失败文件数量和原因。

## 13. Pinecone Key 变更规则

- 修改 Key 后必须重启后端，因为配置和客户端实例会缓存。
- 新 Key 能访问同一个 Index 和 Host 时，已有向量继续有效。
- 新 Key 指向其他账号、项目或 Index 时，SQLite 原始数据仍保留，但所有知识、记忆和项目代码必须重新索引。
- Key 无效时，语义检索降级到 FTS；向量写入和重建操作失败。
- 新 Index 的维度必须与固定 Embedding 模型输出维度一致。

## 14. 迁移与兼容

数据库迁移只新增表和可空字段，不删除现有会话、知识库、记忆和向量记录。旧会话的 `project_id` 保持为空，归入通用对话。

现有 DashScope 配置自动转为默认模型供应商和默认模型配置。现有知识库与长期记忆继续工作。

## 15. 测试与验收

后端至少覆盖：

- 路径规范化、目录逃逸、敏感文件过滤和符号链接边界。
- 增量扫描、新增、修改和删除文件。
- Python AST、FastAPI 路由和 JavaScript 模块解析。
- 多项目数据与 Pinecone namespace 隔离。
- 四类分析制品状态和源码版本失效逻辑。
- 模型供应商配置、密钥隐藏、切换和双模型异常隔离。
- SSE Token 在完整回答结束前到达。
- 断开、取消和失败消息状态。
- 现有知识、记忆、对话和诊断功能回归。

前端至少覆盖：

- 项目添加、切换、重新扫描和删除。
- 最近对话按项目切换。
- 项目分析分组和四类制品页面。
- 模型选择、对比和流式增量渲染。
- 运行状态底部入口。
- 平台 API 文档入口已删除。
- 桌面和移动端输入区、侧边栏和长内容滚动。

完整验收流程：添加当前项目路径，完成扫描，查看项目总览，生成四类制品，进行带源码引用的项目问答，切换两个模型进行对比，确认回答实时流式出现，然后重新扫描修改过的项目并确认旧制品被标记为过期。

## 16. 实施顺序

1. 数据库迁移、项目领域模型、项目源和项目 CRUD。
2. 安全扫描、解析器注册表、结构事实和增量扫描。
3. 项目 FTS、Pinecone namespace 和项目级检索。
4. 四类分析制品生成、保存、过期和展示。
5. 模型供应商、密钥仓库、模型切换和双模型对比。
6. LangGraph、应用用例和 SSE 真流式输出改造。
7. 侧边栏、项目工作区、诊断抽屉和模型设置页面。
8. 数据迁移、自动测试、浏览器端到端验证和启动说明。

实施以完成一条可运行的端到端路径为优先，不在第一版引入独立服务或非必要基础设施。

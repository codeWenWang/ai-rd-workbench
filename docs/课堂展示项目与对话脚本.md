## 2. 本地项目：研发任务管理演示

### 2.1 为什么适合做主演示

这是为课堂演示专门整理的 Java 17、Maven、Spring Boot 多模块项目，业务是容易理解的“研发任务管理”。

- `api`：REST 接口、统一响应、异常处理和应用入口。
- `core-service`：查询、创建、更新、完成和删除任务的业务规则。
- `data-repository`：仓储接口和内存实现。
- `domain`：任务、状态、请求对象和响应对象。

生成的架构图会显示 `api -> core-service -> data-repository -> domain` 的依赖关系；流程图和时序图会以 `GET /api/tasks` 为代表链路；接口文档包含 GET、POST、PUT、DELETE 的 6 个规范接口。

### 2.2 连接与扫描

1. 在项目选择器中点击添加项目。
2. 选择“本地文件夹”。
3. 项目名称填写“研发任务管理演示”。
4. 路径填写 `E:\桌面\AI赋能平台\backend\tests\fixtures\sample_project`。
5. 保存后点击“扫描项目”。
6. 依次打开架构图、流程图、时序图和接口文档。

### 2.3 推荐展示对话

按以下顺序提问，能从项目概览逐步进入源码细节：

1. `请用三句话概括这个项目解决什么问题，并说明 api、core-service、data-repository、domain 四个模块各自负责什么。`
2. `从 GET /api/tasks 开始，按照 Controller、Service、Repository、Domain 的顺序说明完整调用链，并引用对应源码。`
3. `创建研发任务需要提交哪些字段？请给出 POST /api/tasks 的请求样例和响应样例。`
4. `更新任务和完成任务有什么区别？分别对应哪个接口和哪个业务方法？`
5. `如果查询一个不存在的任务 ID，异常从哪里抛出，又在哪里转换成统一响应？`
6. `如果以后要把内存存储替换成 MySQL，最少需要修改哪些模块？为什么？`

### 2.4 单独运行示例项目

在示例目录执行：

```powershell
mvn -pl api -am spring-boot:run
```

然后访问 `http://127.0.0.1:8080/api/tasks`。

## 3. GitHub 项目：springboot-multimodule-example

项目地址：[alexmarqs/springboot-multimodule-example](https://github.com/alexmarqs/springboot-multimodule-example)

### 3.1 为什么选择它

这个项目规模小、模块命名直观，当前平台生成的图不会过密：

- `web` 依赖 `core`、`mail-lib` 和 `security-lib`。
- `core` 依赖 `common`。
- 代表接口为 `GET /v1/customer/admin`。
- 流程图和时序图能够同时展示核心、安全和邮件三个模块。

它只有两个示例接口，因此更适合讲“模块化设计”，不适合替代本地项目做完整 CRUD 教学。

### 3.2 连接与扫描

1. 添加项目时选择“GitHub”。
2. 项目名称填写“GitHub 多模块示例”。
3. 仓库地址填写 `https://github.com/alexmarqs/springboot-multimodule-example.git`。
4. 保存并点击“扫描项目”。
5. 如果本机访问 GitHub 较慢，先确认 Git 代理可用，再重新扫描。

### 3.3 推荐展示对话

1. `请概括这个项目的五个模块，并说明它们之间的依赖关系。`
2. `为什么 common、core、mail-lib、security-lib 要拆成独立模块？这样设计有什么收益？`
3. `从 GET /v1/customer/admin 开始说明请求的处理链路，并引用 CustomerController 的源码。`
4. `guestEndpoint 和 adminEndpoint 的访问控制有什么区别？相关安全配置在哪里？`
5. `如果要增加“创建客户”接口，应该把 Controller、业务逻辑和公共模型分别放在哪些模块？`

## 4. Gitee 项目：mall

项目地址：[macrozheng/mall](https://gitee.com/macrozheng/mall)

### 4.1 为什么选择它

这个项目比其他 Gitee 候选更适合展示真实工程：模块职责明确，且平台能稳定识别出模块依赖和业务接口。

- `mall-admin`：后台管理入口。
- `mall-portal`：用户侧商城入口。
- `mall-mbg`：数据模型和 MyBatis Generator 代码。
- `mall-security`：认证授权能力。
- `mall-common`：公共基础能力。
- `mall-search`：搜索能力。
- `mall-demo`：功能示例。

当前平台生成的代表流程是 `POST /cart/add`，可以直观看到客户端、`OmsCartItemController.add`、`mall-mbg`、`mall-security` 和 HTTP 响应之间的关系。接口文档数量较多，课堂上不建议从头滚动，应使用浏览器页内查找定位 `/cart/add`、`/order` 或 `/product`。

### 4.2 连接与扫描

1. 添加项目时选择“Gitee”。
2. 项目名称填写“Gitee Mall 电商项目”。
3. 仓库地址填写 `https://gitee.com/macrozheng/mall.git`。
4. 保存并点击“扫描项目”。
5. 扫描完成后先展示架构图，再展示“加入购物车”流程图和时序图，最后打开接口文档。

### 4.3 推荐展示对话

1. `请用面向初学者的方式介绍这个电商项目，并说明 mall-admin、mall-portal、mall-mbg、mall-security、mall-common 各自负责什么。`
2. `从 POST /cart/add 开始说明加入购物车的处理链路，并引用 Controller 和相关源码。`
3. `mall-admin 和 mall-portal 有什么区别？它们为什么都依赖 mall-mbg 和 mall-security？`
4. `CommonResult 的统一响应结构是什么？请结合一个实际接口给出请求和响应样例。`
5. `请找出商品、购物车和订单相关的主要 Controller，并分别说明职责。`
6. `如果要增加“购物车商品数量上限”的规则，最适合在哪一层实现？请给出依据。`

## 5. 推荐课堂展示顺序

建议控制在 15 到 20 分钟：

1. 用本地项目完整演示“连接项目 -> 扫描 -> 架构图 -> 流程图 -> 时序图 -> 接口文档 -> 项目问答”。
2. 切换 GitHub 项目，重点比较模块依赖和安全模块拆分。
3. 切换 Gitee 项目，展示平台面对真实大型项目时的扫描能力和业务链路分析。
4. 回到本地项目，提问“替换 MySQL 需要改哪些模块”，展示基于项目源码的扩展建议。

## 6. 上课前检查清单

- 本地服务状态显示正常。
- 默认模型能够正常回答普通问题。
- 三个项目都已提前连接并完成扫描。
- 三个项目的架构图、流程图、时序图和接口文档均已至少打开一次。
- GitHub 网络或 Git 代理已经验证。
- 项目问答前确认左上角选中的是目标项目，而不是通用工作区。
- 大型 Gitee 项目的接口文档使用 `Ctrl + F` 定位具体路径。

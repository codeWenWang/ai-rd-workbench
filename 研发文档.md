## 1. 项目概述

### 1.1 项目背景

AI 研发赋能平台旨在实现标准化的 AI 研发工作流，并构建面向团队的智能问答知识库系统。系统将 AI 能力深度融入研发流程，通过混合检索架构与长短期记忆引擎，提升团队知识复用效率与问答准确性。

### 1.2 核心目标

| 目标 | 说明 |
|------|------|
| 团队知识库 | 支持文本/PDF 入库，混合检索召回团队共享知识 |
| 会话短期记忆 | 滑动窗口 + 摘要压缩，保障多轮对话上下文连贯 |
| 会话长期记忆 | 用户主动录入个性化信息，语义相似度精准召回 |
| 智能问答 | 融合 RAG、长期记忆、短期记忆，生成准确专业回答 |

### 1.3 技术选型

| 类别 | 技术 | 用途 |
|------|------|------|
| 后端框架 | FastAPI + Uvicorn | REST API 服务 |
| 文本模型 | 阿里云百炼 `qwen-plus` | 摘要生成、查询改写、答案生成 |
| 向量模型 | 阿里云百炼 `text-embedding-v4` | 文本向量化（1024 维） |
| 向量数据库 | Pinecone | 长期记忆 & 知识库向量存储 |
| 缓存/短期记忆 | Redis 5.0（Windows 本地） | 会话消息、BM25 语料索引 |
| 混合检索 | rank-bm25 + jieba | BM25 关键词检索 |
| PDF 解析 | pypdf | PDF 文本提取 |
| 前端 | HTML + CSS + JavaScript | 智能问答、记忆录入、知识库管理 |

## 3. 核心模块设计

### 3.1 会话短期记忆（ShortTermMemoryService）

**机制**：滑动窗口 + LLM 摘要压缩  
**存储**：Redis  
**文件**：`backend/app/services/short_term_memory.py`

#### Redis Key 设计

| Key | 类型 | 说明 | TTL |
|-----|------|------|-----|
| `stm:{session_id}:messages` | List | 最近 N 条对话消息（JSON） | 7 天 |
| `stm:{session_id}:summary` | String | 历史对话压缩摘要 | 7 天 |

#### 工作流程

1. 每条消息通过 `RPUSH` 追加到消息列表
2. 当消息数超过 `SHORT_TERM_SUMMARY_THRESHOLD`（默认 8）时，触发摘要压缩：
   - 取出超出滑动窗口的旧消息
   - 调用 `qwen-plus` 生成摘要
   - 合并已有摘要，更新 summary key
   - 裁剪消息列表，保留最近消息
3. 当消息数超过 `SHORT_TERM_WINDOW_SIZE`（默认 10）时，执行 `LTRIM` 保留最新 N 条

#### 上下文组装

```
[历史摘要]
{summary}

user: {message}
assistant: {message}
...
```

---

### 3.2 会话长期记忆（LongTermMemoryService）

**机制**：主动录入 + 语义相似度召回  
**存储**：Pinecone（namespace: `ltm`）  
**文件**：`backend/app/services/long_term_memory.py`

#### 录入流程

1. 接收文本或 PDF 输入
2. PDF 通过 `pypdf` 提取纯文本
3. 语义分块（`semantic_chunk`）
4. 批量向量化（`text-embedding-v4`，1024 维）
5. Upsert 至 Pinecone，附带 `session_id` 元数据
6. 同步更新 Redis 中的 BM25 语料索引

#### 召回流程

1. 使用 `qwen-plus` 对查询进行改写
2. 执行混合检索（BM25 + 向量，RRF 融合）
3. 按 `session_id` 过滤，仅返回当前会话的记忆
4. 格式化返回 Top-K 记忆片段

---

### 3.3 RAG 知识召回（RAGService）

**机制**：查询改写 + 语义分块 + BM25/向量混合检索  
**存储**：Pinecone（namespace: `rag`）  
**文件**：`backend/app/services/rag_service.py`

#### 语义分块策略（DocumentProcessor）

- 优先按段落（`\n\n`）切分，保持语义完整
- 超长段落按句子边界（`。！？.!?`）二次切分
- 默认块大小 500 字符，重叠 50 字符
- 支持 PDF 全文提取后分块

#### 混合检索算法

采用 **Reciprocal Rank Fusion (RRF)** 融合两路召回结果：

```
RRF_score(d) = Σ 1 / (k + rank_i(d))    # k = 60
```

| 召回通道 | 实现 | 分词 |
|---------|------|------|
| 向量检索 | Pinecone `query()` | — |
| BM25 检索 | rank-bm25 + Redis 语料 | jieba 搜索引擎模式 |

BM25 语料存储在 Redis：
- `bm25:{namespace}:corpus` — 分词后的语料列表
- `bm25:{namespace}:ids` — 对应的文档 ID 列表

---

### 3.4 LLM 服务（LLMService）

**模型**：`qwen-plus`（阿里云百炼 OpenAI 兼容接口）  
**文件**：`backend/app/services/llm_service.py`

| 方法 | 用途 | temperature |
|------|------|-------------|
| `summarize()` | 短期记忆摘要压缩 | 0.3 |
| `rewrite_query()` | RAG/长期记忆查询改写 | 0.2 |
| `generate_answer()` | 综合上下文生成最终回答 | 0.7（默认） |

#### API 调用方式

```python
client = OpenAI(
    api_key=DASHSCOPE_API_KEY,
    base_url="https://dashscope.aliyuncs.com/compatible-mode/v1",
)
```

## 9. 关键设计决策

### 9.1 为何使用 1024 维向量

`text-embedding-v4` 支持 64~2048 多种维度，默认 1024。选用 1024 维是因为：
- 性能与成本的最佳平衡点
- 与现有 Pinecone 索引维度匹配
- 阿里云官方推荐的通用检索维度

### 9.2 为何 BM25 语料存 Redis 而非 Pinecone

Pinecone Serverless 索引不原生支持 BM25 稀疏检索。本项目采用：
- Pinecone 负责向量检索
- Redis 存储 BM25 分词语料
- RRF 算法融合两路结果

此方案轻量、无额外中间件依赖，适合中小规模知识库。

### 9.3 长期记忆按 session_id 隔离

长期记忆绑定会话 ID，确保不同用户/会话的个性化记忆互不干扰。召回时先混合检索，再按 `session_id` 元数据过滤。

### 9.4 短期记忆不写入 Pinecone

短期记忆生命周期短（7 天 TTL）、更新频繁，Redis List + 摘要的模式比向量检索更高效、成本更低。

## 11. 常见问题

### Q1：Redis 连接失败 `NOAUTH Authentication required`

Redis 配置了密码但未在 `.env` 中设置 `REDIS_PASSWORD`。检查 `redis.windows.conf` 中的 `requirepass` 配置。

### Q2：Pinecone 向量维度不匹配

确保 `EMBEDDING_DIMENSION=1024` 与 Pinecone 索引维度一致。若索引为其他维度，需重建索引或调整 embedding 维度参数。

### Q3：API 返回 404

确认 API 路由注册在 StaticFiles 挂载之前。`/api/*` 路由必须在 `app.mount("/", StaticFiles(...))` 之前定义。

### Q4：PDF 入库后检索不到内容

- 确认 PDF 包含可提取文本（非扫描件/图片 PDF）
- 检查 Pinecone 控制台对应 namespace 是否有向量写入
- 尝试更具体的问题触发 RAG 召回

### Q5：中文 BM25 分词效果

项目使用 `jieba.cut_for_search` 进行中文分词。首次运行会构建 jieba 词典缓存（约 0.5~1 秒）。
